# 2\. 背景与核心理念

传统 AI 旅游助手多采用「纯文本聊天」模式，容易产生冗长回复，用户认知负荷大，且缺乏对行程安排的直观掌控感。本产品反其道而行：**把对话降级为低频的意图捕捉工具，把结构化看板（Schedule）、卡片流（Feed）和地图（Map）作为主交互**，让用户直观看到推荐、并掌握最终行程的编排权。

**传统模式痛点**

- 长文本输出，信息密度低

- 推荐不可视，难以横向比较

- AI 内容「信任危机」，缺溯源

- 行程不可拖拽编排，掌控感弱

**本产品主张**

- 卡片化推荐，滑动即比选

- 推荐附来源链接，可追溯

- 日程表 \+ 实时地图双栏联动

- 用户拖拽编排，AI 辅助而非替代决策

## 2\.1 设计原则

1. **可视优于文字**：能用卡片/地图/时间轴表达的，绝不堆文字。

2. **溯源即信任**：所有推荐可追溯到来源链接。

3. **用户掌舵**：AI 出草案，用户拥有增删改拖与撤销的最终编排权。

4. **决策成本最小化**：把交通、耗时等关键决策信息前置到卡片上。

5. **单一数据源（Single Source of Truth）**：行程状态唯一权威来源，卡片流/日程表/地图均为其投影（详见 §8\.4）。

# 5\. 功能需求详情

**说明：**P0=MVP 必须；P1=首期应有；P2=可延后。**MVP 主链路仅依赖高德 POI 自带数据跑通**，比价与内容抓取均延后。

## 5\.1 初始化漏斗：混合输入的决策分流

行程前通过**「对话输入框 \+ 结构化选项卡」混合输入**确认优先级：自然语言意图（如「成都耍三天，爱吃辣」）由对话框捕捉，目的地/天数/预算等结构化条件由选项卡快速点选，二者结果合并为规划输入。

### 5\.1\.1 路径 A：交通 / 时间优先（P1，延后）

**注意：**本路径依赖 OTA 比价数据接入（需资质），**明确延后至 P1**。功能点：聚合机票/高铁、多平台比价、展示行李额/飞行时间/红眼标识；亮点：错峰日期更优价组合。

### 5\.1\.2 路径 B：目的地 / 体验优先（P0，MVP 主链路）

- **核心逻辑**：先定玩什么，再定怎么去。**\[P0\]**

- **功能点**：用户输入目的地/体验 → AI 结合**高德 POI 数据**推荐核心景点 → 基于地理位置给出「不走回头路」顺路线路 → 选定后匹配大交通（大交通比价部分随 §5\.1\.1 延后，先给方向性建议）。**\[P0\]**

## 5\.2 重感知、强交互的推荐系统

### 5\.2\.1 精选卡片流（Carousel Feed）

- 吃、住、玩横向滚动卡片；手势/拖拽浏览，点击勾选（加入行程）或剔除（重新生成该类目）。**\[P0\]**

### 5\.2\.2 内容溯源与信任背书（调整方案）

**推荐（合规方案）：**v1 **不做小红书/抖音定向抓取**，改为使用**通用 fetch / Web 搜索工具**直接检索公开网页，引用**公开链接 \+ 简短摘要**，规避平台 ToS 与版权风险。

- 推荐语限制 **50 字以内**（字符计数规则见 §11\.3）；**\[P0\]**

- 每项推荐附 1 条可点击的公开来源链接；**\[P0\]**

- 定向平台抓取与高赞评论解析**明确延后**（依赖反爬与合规）。**\[P2/延后\]**

### 5\.2\.3 酒店 / 机票透明化（P1，延后）

**注意：**OTA 多平台实时比价、含行李额/退改签/红眼标识等，**依赖外部资质，延后至 P1**。MVP 阶段酒店/景点信息以高德 POI 自带字段为准。

## 5\.3 双栏联动：实时渲染地图 \+ 日程表（Web 左右分栏）

### 5\.3\.1 左侧：动态日程表

- 时间轴展现每日行程节点；**\[P0\]**

- **增量智能计算**：增删/拖拽地点时，**仅重算受影响的相邻交通段**（非全量 O\(N²\)），配合**本地缓存 \+ 防抖（debounce）**降低高德 API 调用量；**\[P0\]**

- **时间预估**：预估交通耗时与建议停留，时间轴展示空闲间隙或冲突重叠。**\[P1\]**

### 5\.3\.2 右侧：实时渲染地图（Live Map）

- **动态连线**：挑选/挪动地点，地图实时打点并渲染轨迹；**\[P0\]**

- **交通可视化**：按所选小交通展示对应路线（步行/驾车），叠加实时路况；路线以**颜色 \+ 线型 \+ 图例**区分（无障碍）。**\[P1\]**

### 5\.3\.3 关键边界与异常场景

|场景|预期处理|优先级|
|---|---|---|
|空行程（无 POI）|展示空状态引导，提示从卡片流添加|P0|
|跨城 POI（不同城市）|识别为多城行程，分城分段展示，提示大交通衔接|P1|
|海上 / 无路网点|无法路径规划时，地图仅打点不连线，标注「无可达路线」|P1|
|POI 经纬度缺失|降级为「仅列表展示」，地图不打点并提示坐标缺失|P0|
|单日 POI \> 15（点聚合临界值）|触发地图点聚合（Clustering），日程分段折叠|P1|
|同时段地点冲突|时间轴高亮重叠并提示调整|P1|
|地图/比价接口超时|降级展示缓存数据 \+ 友好提示（见 §8\.3）|P0|

## 5\.4 账号与登录（P0）

**推荐（v1 方案）：第一版必须实现登录**，采用**邮箱 \+ 密码**方式，**无需邮箱验证码**；注册时二次确认密码，校验通过后直接写入数据库。行程数据与账号绑定持久化。

### 5\.4\.1 功能点

- **注册**：输入邮箱 \+ 密码 \+ 二次确认密码；前后端校验邮箱格式与两次密码一致后写库。**\[P0\]**

- **登录**：邮箱 \+ 密码校验，签发会话凭证（如 JWT / Session）。**\[P0\]**

- **登出**：清除会话凭证。**\[P0\]**

- **行程归属**：登录后 Itinerary 与 user\_id 绑定持久化（对应 §10 数据模型）。**\[P0\]**

### 5\.4\.2 安全要求

|项|要求|
|---|---|
|密码存储|禁止明文；使用 bcrypt / argon2 加盐哈希后入库|
|传输|全程 HTTPS|
|密码强度|最小长度 ≥ 8，前端即时校验，二次确认一致方可提交|
|会话|凭证设置合理过期；登出失效|
|防暴力破解|登录失败限频（如 5 次/分钟）|

### 5\.4\.3 验收标准（GWT）

|场景|验收标准|
|---|---|
|注册成功|**Given** 合法邮箱与两次一致密码 **When** 提交注册 **Then** 账号写库（密码哈希存储），自动登录并可进入规划|
|两次密码不一致|**Given** 两次密码不同 **When** 提交 **Then** 前端拦截并提示，不发起请求|
|登录鉴权|**Given** 已注册账号 **When** 正确邮箱密码登录 **Then** 签发凭证，行程与该账号绑定|
|错误密码|**Given** 密码错误 **When** 登录 **Then** 提示失败，连续失败触发限频|

**注意：**v1 暂不做邮箱验证码 / 找回密码 / 第三方 OAuth，作为后续迭代项；上线前需评估「无验证码」下的垃圾注册风险，必要时补图形验证码。

## 7\.3 Agent 编排（Harness）选型

**结论：v1 自研轻量 Workflow 编排，二期视复杂度升级 LangGraph。** Claude Code / Codex 等是面向开发者的「编码 Agent」，并非线上运行时框架，不适合直接作为本产品的编排内核。本产品流程高度确定（意图捕捉 → POI 推荐 → 顺路线路 → 交通计算 → 渲染），更适合确定性 Workflow 而非全自主 Agent loop。

|方案|适配度|说明|
|---|---|---|
|自研轻量 Workflow 编排|⭐ 最推荐|节点少、链路固定，配合 OpenAI 兼容 SDK 即可；可控、易调试、零框架学习成本，最适合单人项目|
|LangGraph|推荐备选|未来做「行中动态调度/异常重规划」等有状态、有分支回环编排时升级；当前 P0 略重|
|全自主 Agent（AutoGPT 类 / 编码 Agent）|❌ 不推荐|不可控、延迟高，与低延迟可视化定位冲突|

**设计要点：**每个编排节点都需**可独立测试、可降级**（对应 §8\.3 与 §12\.5 契约测试），并统一通过 §7\.2 的 OpenAI 兼容接口调模型，使 Claude 4\.8 / DeepSeek V4 Pro 切换不影响编排骨架。

# 8\. 系统架构

## 8\.1 耗时预算拆分（关键修订）

|链路|SLA|说明|
|---|---|---|
|纯数据聚合（地图/POI/比价）|**≤ 2s**|不含 LLM 的同步聚合请求|
|含 LLM 规划链路 \- 首字|**≤ 3s**|流式首 token 返回|
|含 LLM 规划链路 \- 完整草案|**≤ 30s**|完整行程草案渲染完成|
|首份可见内容（TTFP）|**≤ 5s**|骨架屏 \+ 流式填充首个真实内容|

## 8\.2 BFF 缓存策略（粒度定义）

|数据类型|缓存键|TTL|过期展示口径|
|---|---|---|---|
|POI 详情|poi\_id|24h|直接使用|
|两点路径/耗时|起点\_终点\_交通方式|1h|命中即复用，免重复请求|
|实时路况|路段\_视口|5min|过期标注「更新于 X 分钟前」|
|比价（延后）|航线/酒店\_日期|10min|超时展示缓存价 \+ 时间戳提示|
|LLM 推荐结果|意图哈希|会话级|同输入复用，避免重复推理|

## 8\.3 降级策略（粒度定义）

|故障|降级行为|
|---|---|
|地图接口超时/限流\(429\)|展示上次缓存轨迹 \+ 「路况更新失败，显示历史数据」提示|
|POI 搜索返回空|展示空状态，引导调整关键词或扩大范围|
|LLM 超时/失败|降级为高德 POI 热门排序的非个性化推荐|
|返回脏数据（坐标越界/字段缺失）|BFF 层校验过滤，单条丢弃不阻塞整体|
|比价接口超时（延后能力）|展示缓存参考价 \+ 时间戳|

## 8\.4 单一数据源原则（Single Source of Truth）

**原则：**前端维护唯一权威的 **Itinerary Store**（行程状态机）。卡片流、日程表、地图三者**均为该 Store 的只读投影**，任何交互（勾选/拖拽/删除/撤销）都先变更 Store，再由 Store 单向驱动三个视图重渲染，杜绝多视图各自持状态导致的不一致与竞态。

## 13\.3 三态规范（空 / 加载 / 错误）

|状态|设计要求|
